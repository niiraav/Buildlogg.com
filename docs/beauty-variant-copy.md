> **Purpose:** Repurpose Buildlogg's existing landing page design for the beauty/salon vertical. Same HTML structure, same CSS, same PWA — different copy, different pain angle, different demo content.

The design uses Lucide icons, Inter/Manrope fonts, and a phone mockup with animated demo frames. All CSS is shared.
>
> **Task:** Write complete replacement copy for every text element in the landing page, adapted for UK beauty/salon service businesses (nail techs, beauty salons, tattoo artists, lash techs, therapists). The core pain is no-shows and late cancellations — not lost quotes. The hero feature is deposit enforcement + automated reminders, not quote sending.
>
> **Constraints:**
> - Keep the exact same HTML section structure (Nav, Hero, Micro-strip, Problem, How it works, Feature, Secondary features, Pricing, CTA, Footer)
> - Match the tone: direct, conversational, no corporate jargon, UK English
> - Every headline must be under 12 words
> - Every sub-headline must be under 25 words
> - Feature descriptions max 2 sentences
> - The 3-frame phone demo needs new content (beauty booking → deposit → reminder, not quote → send → paid)
> - Pricing stays the same structure (£0 Free, Pro free during beta)
> - Trust badges: "Free to start", "Works offline", "No card needed" → adapt for beauty
> - The brand name stays "Buildlogg" — same domain, same app, different landing page at /beauty/
>
> **Voice:** Like talking to a salon owner who's exhausted from chasing clients who don't show up. Direct but warm. Not salesy.

## How It Works Technically — buildlogg.com/beauty/

### Approach: Route-based Landing Page Variants

The current setup is a Cloudflare Pages deployment with a single `index.html` at the root. The PWA app lives at `/app/`. We need a `/beauty/` route that serves a variant of the landing page.

**Option A — Static HTML variant (recommended, fastest):**
1. Copy `index.html` → `public/beauty/index.html`
2. Replace all copy with the beauty variant above
3. Update demo phone frames with beauty content
4. Update meta tags, OG tags, Twitter cards
5. Update PostHog to track `source=beauty-landing` on signup links
6. Deploy — Cloudflare Pages serves it at `buildlogg.com/beauty/`

No code changes needed. No routing changes. Just a static HTML file in a new directory. The CSS (`/assets/landing.css`) is shared — no CSS changes needed since all classes are reused.

**Option B — Dynamic template (future, when you have 5+ variants):**
Build a single template with JSON-driven copy. One HTML file, `?variant=beauty` changes all text. Overkill for now.

### File Changes Required

| File | Change | Effort |
|------|--------|--------|
| `public/beauty/index.html` | NEW — copy of index.html with beauty copy | 2 hrs |
| `public/beauty/` assets | Reference parent `/assets/landing.css`, `/assets/icon-black-square.png` etc. via absolute paths | 0 |
| PostHog tracking | Add `source=beauty-landing` to all `/app/auth` links | 15 min |
| Email templates | Adapt 4-step cold email sequence for beauty vertical | 1 hr |
| Lead list | Export 200 UK beauty/salon Instagram handles | 1 hr |

### Why This Works

- Same app, same Supabase backend, same auth, same Stripe — zero backend changes
- The PWA doesn't know or care which landing page sent the user
- PostHog `source` param lets you track conversion per variant
- If beauty converts better, you double down. If not, you try another vertical with a new 2-hour landing page
- The domain authority of buildlogg.com benefits all variants

### What NOT to Build Yet

- No separate branding/domain for beauty
- No booking calendar feature (use the existing job/quote flow repurposed as booking/deposit)
- No separate app — same PWA, different entry point
- No email automation changes beyond copy adaptation
